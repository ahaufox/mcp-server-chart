### 背景
为了保证项目文档的实时性和准确性，当代码发生重大逻辑变更、底层架构调整或部署方式改变时，必须同步更新主 `README.md` 文档。

### 触发条件
当任务涉及以下变更时，执行此工作流：
1. **架构变更**：新增或删除核心服务（如新增 Redis 缓存层、更换数据库引擎）。
2. **部署调整**：`docker-compose.yml` 结构发生重大变化，或新增了关键环境变量（`.env`）。
3. **关键逻辑**：后端 `AppService` 或插件管理机制（Plugin Daemon）的底层行为发生改变。
4. **子模块变更**：更新了 Git Submodule 的引用或目录结构。

### 执行步骤

#### 1. 评估变更影响
回顾当前任务所做的所有修改，判断其是否属于上述【触发条件】。

#### 2. 定位更新区域
根据变更类型，定位 `README.md` 中的对应章节：
- `🚀 项目架构`：涉及组件增减或协作逻辑变更。
- `🛠️ 快速开始`：涉及克隆、配置环境或启动命令变更。
- `📦 核心配置`：涉及数据库、网络、安全等关键配置项。
- `📁 目录结构`：涉及源码或配置文件位置变动。

#### 3. 实施更新
使用 `replace_file_content` 或 `multi_replace_file_content` 更新 `README.md`：
- **强制要求**: 除非用户明确要求英文，否则必须**默认使用中文**维护 `README.md`。
- **一致性检查**: 确保更新后 `README.md` 依然保留并对齐 `Agent Intelligence`（AI 智能）章节。
- 保持文档简洁，避免过于琐碎的技术细节（细节应留在 `implementation_plan.md` 中）。
- 使用清晰的 Markdown 语法和 Emoji。
- **重要**：如果新增了部署命令或修复步骤，优先考虑将其固化为单独的 `.agent/workflows/*.md` 文件，并在 `README.md` 中引用。

#### 4. 验证链接
确保 `README.md` 中引用的所有本地文件链接（如 `./LICENSE`）依然有效。

### 角色建议
1. 在每次任务完成前，应主动检查是否需要运行此工作流，而不是等待用户提醒。
2. 维护文档时，始终将自己视为“**首席布道师 (CTO/Chief Evangelist)**”，确保文字具备感染力。
